## 1. Why this module exists

`validateSpawn()` (#99 + #103) returns `{ ok:false, code, detail }` on a G1–G6 / P1 violation. The eventual spawn caller (#15, not yet wired) will translate that into a thrown `SpawnValidationError`. ADR §6 task #11 is the moment that throw becomes user-facing fatal. Between landing #99/#103/#115 today and flipping #11 tomorrow, we need a **compatibility-audit window**: violations surface as telemetry events + console warnings, but the spawn still proceeds with a degraded fallback role so existing dispatches keep working while we measure how many actually trip a gate.

This SPEC adds:

1. A **mode flag** (`'hard-fail' | 'warn' | 'off'`, default `'warn'`) read from env / config.
2. A **policy wrapper** `enforceSpawn(req, opts)` over the existing pure `validateSpawn()` — same module, additive, no edit to G1–G6 / P1 bodies.
3. A **telemetry module** (`src/telemetry/spawn-events.ts`) that emits append-only NDJSON to `~/.aigentry/telemetry/spawn-events-YYYY-MM-DD.ndjson`.
4. An **aggregation script** (`bin/spawn-telemetry-report.sh`) + a `state/telemetry/SUMMARY.md` surface for the orch dashboard.
5. A **≤5-line AGENTS.md note** pinned to the existing Rule 4 row (line 50).

---

## 2. Naming discipline — wrapper layer, not a new gate

`validateSpawn()` stays a pure decision function (returns a result; never throws, never emits). The new entrypoint `enforceSpawn()` is the **policy** layer that maps the pure result onto current operating mode. This keeps #99/#103 byte-for-byte intact (Rule 29) and makes the eventual #15 spawn-path integration a one-line call-site swap. No new G/P labels are introduced.

## 3. Module API surface

### 3.1 Extension to `src/session/validate-spawn.ts` (additive)

```ts
export type ValidationMode = "hard-fail" | "warn" | "off";

export const DEFAULT_VALIDATION_MODE: ValidationMode = "warn";  // §6 #9 window

export interface EnforceSpawnOptions extends ValidateSpawnOptions {
  ctx_digest?: string;            // from #114 ResolvedInstructions when available
  mode?: ValidationMode;          // explicit override (tests); else env-var
  now?: () => Date;               // injectable clock (tests)
  emit?: (e: SpawnEvent) => void; // injectable sink (tests)
}

export type EnforceSpawnResult =
  | { ok: true;  mode: ValidationMode; effective_role: Role; degraded: false; validation: { ok: true } }
  | { ok: true;  mode: "warn";        effective_role: Role; degraded: true;  validation: ValidateSpawnFailure }
  | { ok: false; mode: "hard-fail";   validation: ValidateSpawnFailure };
// `ok: false` callers throw (`SpawnValidationError`); `ok: true` callers proceed.

export function readValidationMode(env?: NodeJS.ProcessEnv): ValidationMode;
// Reads `AIGENTRY_SPAWN_VALIDATION_MODE`. Unknown / unset → DEFAULT_VALIDATION_MODE.

export function enforceSpawn(req: SpawnRequest, opts?: EnforceSpawnOptions): EnforceSpawnResult;
```

Behavior matrix:

| Mode        | validateSpawn ok | validateSpawn fail |
|-------------|------------------|--------------------|
| `hard-fail` | emit `spawn_accepted` → return `{ok:true, degraded:false, effective_role:req.role}` | emit `spawn_rejected` → return `{ok:false, validation}` (caller throws) |
| `warn`      | emit `spawn_accepted` → return `{ok:true, degraded:false, effective_role:req.role}` | emit `spawn_rejected` + emit `spawn_degraded` + `console.warn` → return `{ok:true, degraded:true, effective_role:Role.coder, validation}` |
| `off`       | (skip validation) → no event → return `{ok:true, degraded:false, effective_role:req.role, validation:{ok:true}}` (see §3.4) | (validation not run) |

Constraint: `enforceSpawn` performs **at most one mode read per call** to allow mid-run env mutation; see §3.5.

### 3.2 New module — `src/telemetry/spawn-events.ts`

```ts
export type SpawnEventKind = "spawn_accepted" | "spawn_rejected" | "spawn_degraded" | "mode_changed";

export interface SpawnEvent {
  ts: string;                  // ISO-8601 UTC, microsecond-resolution per ADR §4.8.2
  event: SpawnEventKind;
  mode: ValidationMode;
  session_id: string | null;   // proposed_session_id at call time (may be null pre-snapshot)
  parent_id: string | null;
  reason: string;              // for `spawn_rejected`: ValidateSpawnErrorCode. For `mode_changed`: "prev→curr".
  violations: ReadonlyArray<{ code: string; detail: string }>;
  ctx_digest: string | null;   // ResolvedInstructions digest from #114 when supplied
}

export interface EmitOptions {
  root?: string;               // override for tests; default os.homedir()/.aigentry/telemetry
  now?: () => Date;
}

export function emit(event: SpawnEvent, opts?: EmitOptions): void;
export function telemetryRoot(opts?: { root?: string }): string;
export function dailyFilePath(date: Date, opts?: { root?: string }): string;
export function readEventsForDay(date: Date, opts?: { root?: string }): SpawnEvent[];  // helper for tests + report
```

**File layout (Article 2 cross-platform):**

```
${os.homedir()}/.aigentry/telemetry/spawn-events-YYYY-MM-DD.ndjson
```

- Date suffix is **UTC** date of `ts` — natural daily rotation, no separate rotator process.
- Dir created on first emit with `fs.mkdirSync(..., { recursive: true })`.
- Append via `fs.appendFileSync(path, JSON.stringify(event) + "\n", { encoding: "utf8", flag: "a" })` — atomic per-line on POSIX; Windows append acceptable per Node docs.
- One event = one line. No nesting. `JSON.parse` round-trip is identity (§4 invariant).

**Privacy filter (mandatory):** `emit()` rejects (`throw TypeError`) any event object containing keys outside the schema above. Test §4.8 asserts this. The schema explicitly excludes: task body, effective_prompt text, cwd path (already in dispatch context; not a user secret but also unnecessary for audit signal), permission-set contents (kept abstract via reason code).

### 3.3 New helper — `bin/spawn-telemetry-report.sh`

```
spawn-telemetry-report.sh [--days N] [--root DIR] [--out PATH]
```

- Default: scan last 7 daily NDJSON files under `~/.aigentry/telemetry/`, write markdown to `state/telemetry/SUMMARY.md`.
- Markdown sections:
  - Total counts by event (accepted / rejected / degraded / mode_changed).
  - Top 5 rejection reasons (group by `reason`).
  - Top rejected roles (derived: parse reason → resolve to requested role via `violations[].detail` heuristics; fallback "unknown").
  - Mode-transition log (every `mode_changed` event with ts).
- ≤80 LOC. Implementation: `awk` / `jq` if present, else pure POSIX `sh` + Node one-liner fallback for JSON parse (we already require Node ≥20 in `package.json`, so `node -e "..."` is acceptable). Decision: use `node -e` to keep parsing strictly correct — no `jq` dependency.

### 3.4 `'off'` mode semantics

When `mode === 'off'`:

- `validateSpawn()` is **not invoked** (escape-hatch latency win).
- No per-spawn telemetry is emitted (volume kept at zero — emergencies should not flood the log).
- A **single** `mode_changed` event is emitted the first time `'off'` is observed in a process (covered by §3.5). This is the "logged" requirement from the dispatch.

### 3.5 Mode-transition detection

Module-scope state in `validate-spawn.ts`:

```ts
let _lastObservedMode: ValidationMode | undefined;

function trackMode(current: ValidationMode, parent_id: string | null, emitFn: (e: SpawnEvent) => void, now: () => Date): void {
  if (_lastObservedMode !== undefined && _lastObservedMode !== current) {
    emitFn({
      ts: now().toISOString(),
      event: "mode_changed",
      mode: current,
      session_id: null,
      parent_id,
      reason: `${_lastObservedMode}→${current}`,
      violations: [],
      ctx_digest: null,
    });
  }
  _lastObservedMode = current;
}

// Test-only:
export function __resetModeTrackingForTests(): void { _lastObservedMode = undefined; }
```

Invariants:
- First call after process boot sets the latch but emits nothing (no prior state).
- Subsequent observed mode differing from latched value emits exactly one `mode_changed`.
- Reset hook exists strictly for unit test isolation (named `__` per Article 17 / TS-strict convention).

## 4. Test plan — `tests/session/warn-mode.test.ts` (≥ 10 scenarios)

Test runner = `node:test` (matches existing `tests/session/*.test.ts`). All tests inject `emit` + `root` + `now` so nothing touches the real `~/.aigentry/` during CI.

1. **warn-mode, G1 violation** — missing role → `emit` records `spawn_rejected` + `spawn_degraded`, result is `{ok:true, degraded:true, effective_role:"coder"}`.
2. **warn-mode, G4 violation** — non-absolute cwd → degraded fallback as above; `console.warn` invoked (spy via `t.mock.method(console, "warn")`).
3. **warn-mode, P1 violation** — child requests `network` capability the parent lacks → degraded fallback; `reason === "ERR_CAPABILITY_DENIED"`.
4. **hard-fail mode, violation** — same G1 input as test 1 → `{ok:false, mode:"hard-fail"}`; one `spawn_rejected` event; `console.warn` NOT called.
5. **hard-fail mode, happy path** — valid request → `{ok:true, degraded:false, effective_role:req.role}`; one `spawn_accepted` event.
6. **warn-mode, happy path** — valid request → `{ok:true, degraded:false}`; one `spawn_accepted` event only (no degraded).
7. **off mode, would-violate input** — invalid input → `{ok:true, degraded:false, effective_role:req.role}`; **zero** spawn_accepted / spawn_rejected events.
8. **off mode + mode transition** — sequence `warn` → `off` → `warn`: two `mode_changed` events emitted (warn→off, off→warn); no other events while in `off`.
9. **Cross-day rotation** — inject `now` returning UTC dates spanning midnight → events split across `spawn-events-2026-05-12.ndjson` and `spawn-events-2026-05-13.ndjson`; both files exist; per-day `readEventsForDay()` returns the correct slice.
10. **NDJSON round-trip + schema validation** — emit 5 mixed events, read back via `readEventsForDay()`, deep-equal assert; each parsed event matches the `SpawnEvent` schema (no extra / missing fields).
11. **Privacy guard** — `emit({ ...evt, task_body: "secret" } as any)` throws `TypeError`; the file is not written.
12. **Report aggregation** — seed 15 events across 2 days, run `bin/spawn-telemetry-report.sh --root <tmp> --out <tmp/out.md>`, assert markdown contains expected counts + top-reason ranking.

Estimated test LOC: ~220 (≤ 250 ✓).

## 7. Backwards compatibility

- `validateSpawn()` signature + return shape: **unchanged**. All existing #99 / #103 / #115 tests pass without edits.
- Existing tests that need pure validateSpawn behavior continue to call `validateSpawn()` directly (no mode involved).
- The new `enforceSpawn()` is the only mode-aware surface. Its default mode is `'warn'` — when #15 wires it into the spawn path, violations turn into warnings, not crashes.
- Hard-fail-style tests for #99/#103 (if added later) call `enforceSpawn(req, { mode: 'hard-fail' })` explicitly.

## 8. Open questions for orchestrator review

1. **Default mode confirmation** — dispatch says `'warn'` is the compatibility-window default. Confirm this is the intended ship-state of #9 (not `'off'` for the very first PR).
2. **Degraded role choice** — dispatch fixes fallback to `Role.coder`. If the original request was `tester` / `analyst`, demoting to `coder` may grant *more* default capabilities (per §4.6.2 table) than the violator asked for. Should the fallback role be the **intersection-safe** `logger` (minimum capability set) instead of `coder`? — I lean toward `logger` for safety; will await orch decision.
3. **`spawn_accepted` volume** — emitting on every successful spawn doubles the audit signal but inflates the NDJSON file. Alternative: emit `spawn_accepted` only in `'warn'` mode (where audit matters) and skip in `'hard-fail'` (where success is silent). Default proposal: emit always; orch may downgrade.
4. **`hard-fail` mode throws vs. returns** — current dispatch text says "throw"; my SPEC returns `{ok:false}` and lets the caller (eventually #15) throw. This keeps `enforceSpawn` testable without try/catch but diverges from the wording. Confirm whether the wrapper itself should `throw new SpawnValidationError(...)` directly.
5. **Daily file rotation timezone** — UTC date suffix chosen for determinism. If the orch dashboard reads in local time, mid-day-local cross-over could split a session's events across two files. UTC is the right default for distributed correctness, but flag if local-time grouping is preferred for the SUMMARY.md aggregator.
6. **`bin/spawn-telemetry-report.sh` parser dependency** — proposing `node -e` fallback to avoid a `jq` runtime requirement (Article 17). Confirm acceptable.
